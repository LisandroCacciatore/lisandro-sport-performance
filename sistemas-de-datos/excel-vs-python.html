Cuándo migrar de Excel a Python en datos deportivos: señales concretas

El techo de Excel en datos deportivos no es un problema de hardware. Es un problema de arquitectura. La migración a Python se justifica cuando aparecen cuatro señales específicas — y la pregunta clave antes de migrar no es "¿sabemos programar?" sino "¿tenemos claro qué queremos hacer con los datos?"

Lisandro Cacciatore · Sistemas de Datos

El problema no es Excel. Es el uso que le damos.

Excel es una herramienta de cálculo. No es una base de datos, no es un pipeline y no fue diseñado para gestionar series temporales de equipos deportivos con tres fuentes de datos simultáneas. Cuando lo usamos para eso, el resultado siempre es el mismo: el archivo crece, se rompe, y alguien termina reconstruyendo columnas a las 11PM del viernes.

He visto archivos llamados MASTER_V3_FINAL_OK.xlsm que pesan 85MB, tienen 14 pestañas con referencias cruzadas y colapsan si alguien cambia una celda sin avisar. No es un problema de quién los armó. Es un problema de escala.

Las cuatro señales de que el proceso ya no escala

Hay señales concretas de que llegaste al techo:

  • El archivo tarda más de 30 segundos en abrir o calcular.
  • Necesitás comparar carga de hoy contra el promedio de los últimos tres martes de microciclo largo, y eso exige una fórmula de tres líneas que nadie más entiende.
  • Agregaron dos jugadores y alguien tuvo que "ajustar" las fórmulas manualmente.
  • Hay más de una versión del archivo circulando por WhatsApp.

Cuando aparece alguna de estas señales, el problema no es que falta una macro más. Es que la estructura de datos y la estructura de decisión quedaron desconectadas.

La migración no es aprender a programar

La solución no es reemplazar Excel con software complejo ni contratar un equipo de ingeniería. Es usar Python como un proceso de automatización simple: leer, limpiar, validar y exportar. Sin intervención manual.

Este script de cinco líneas hace lo que cuatro horas de copy-paste intentan hacer:

pipeline_basico.py 
import pandas as pd
import glob

# 1. Leer todos los CSVs de la sesión sin abrir Excel
archivos = glob.glob('data/gps/*.csv')
df = pd.concat((pd.read_csv(f) for f in archivos))

# 2. Filtrar ruido (GPS prendido pero sin sesión real)
df_clean = df[df['Duration_Min'] > 20]

# 3. Exportar base limpia lista para Power BI o Sheets
df_clean.to_csv('base_validada.csv', index=False)

Lo que cambia no es la tecnología. Es que el proceso ya no depende de que alguien lo ejecute correctamente a mano. Si la lógica está bien definida en el script, se aplica igual todos los días, para todos los jugadores, sin importar quién está en la oficina.

Qué gana el club con esto

Trazabilidad: si hay un error, se ubica en qué línea del proceso ocurrió, no en qué pestaña de qué versión del archivo. Velocidad: procesar una temporada entera lleva segundos. Escalabilidad: si entran cinco jugadores nuevos, el script no lo nota.

Y lo más importante: el analista deja de ser el guardián manual del archivo y pasa a ser quien interpreta los resultados. Eso es lo que le agrega valor al proceso de decisión.

La pregunta que importa antes de migrar

No es "¿sabemos programar?" sino "¿tenemos claro qué queremos hacer con los datos?". La sintaxis de Python se aprende o se delega. Lo que no se puede delegar es definir qué pregunta tiene que responder el sistema.

Un club que no sabe qué preguntas le hace al dato no va a resolverlo con Python. Va a tener el mismo problema, con mejor código.

Preguntas frecuentes

¿Cuándo Excel ya no alcanza para gestionar datos deportivos? +

El límite aparece cuando el archivo tarda más de 30 segundos en calcular, cuando las fórmulas necesitan más de una línea para un cruce simple, cuando hay más de una versión circulando por WhatsApp, o cuando agregar un jugador nuevo requiere ajuste manual. Cualquiera de estas señales indica que la estructura de datos y la de decisión quedaron desconectadas.

¿Cuánto sabe de programación necesita un analista deportivo para usar Python? +

La migración no requiere que el analista se vuelva programador. Requiere que tenga claro qué quiere hacer con los datos. La sintaxis de Python se aprende o se delega. Lo que no se puede delegar es definir qué pregunta tiene que responder el sistema.

¿Qué ventajas tiene Python sobre Excel para datos GPS? +

Tres ventajas estructurales: trazabilidad (el error se ubica en qué línea del proceso, no en qué pestaña), velocidad (una temporada entera en segundos), y escalabilidad (cinco jugadores nuevos no requieren ajuste). La más importante: el analista deja de ser guardián manual del archivo y pasa a interpretar resultados.

¿Por qué un club sin preguntas claras no se beneficia de Python? +

Porque Python automatiza un proceso, no lo diseña. Si el proceso no tiene lógica clara, la automatización solo hace más rápido algo que no sirve. La migración más eficiente empieza por definir las preguntas del cuerpo técnico, no por elegir tecnología.
